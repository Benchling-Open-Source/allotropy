Agilent Gen5 Plate Reader Adapter Requirements

Revision 05

Date: 2024-04-29

The Agilent Gen5 Plate Reader adapter is intended to convert text file exports from the Gen5 software and generate a .json file corresponding to the plate-reader Allotrope Simple Model (ASM). The Gen5 software supports the complete product line of BioTek brand plate readers manufactured by Agilent, which includes single-mode and multi-mode plate readers. These instruments are able to perform measurements using one (single) or more (multi) detection modality including: absorbance, fluorescence, luminescence, optical-imaging measurements. The plate-reader ASM supports the recording of data from these different measurement modalities by referencing separate measurement schema for uv-absorbance, fluorescence, luminescence, or optical-imaging.

The adapter specified in this requirements document will contain requirements for handling of absorbance, fluorescence, and luminescence measurements from Gen5. Handling of optical-imaging measurements has been separated into a different adapter requirements, here. The adapter is expected to recognize the detection modality used to generate the data and generate the plate-reader ASM .json file using the appropriate measurement schema (absorbance, fluorescence, luminescence).

The adapter is expected to produce the following output based on the provided input:

Agilent Gen5 Example Files

The ASM model schema are detailed in the following:

  • plate-reader ASM schema
  • ultraviolet-absorbance-point-detection ASM schema
  • fluorescence-point-detection ASM schema
  • luminescence-point-detection ASM schema

The default Gen5 export file is a .TXT file with a number of ‘blocks’ which contain relevant measurement and metadata required for the ASM. These include a “Header”, Layout, and Results.

The ‘blocks’ follow the pattern of:

Header:

– Contains a number of keyword:value pairs recording metadata about the instrument, software, and measurement performed

Layout:

– Contains details related to the samples being measured. The Layout block is formatted as a plate matrix (row x column) but with every position within the matrix functioning as a list of entries - with the label for each entry in the list appearing in the right margin.

Results:

– Contains both measurement values and calculated results corresponding to an individual well within the plate. The Results block is formatted similarly to the Layout block as a plate matrix (row x column) again with every position in the matrix functioning as a list of entries - with the label for each entry in the list appearing in the right margin

 

Note: The Gen5 adapter is only expected to support the Default export format, which follows this Header/Layout/Results pattern. The Default export format can be discerned by the keywords: Software Version, Experiment File Path, Protocol File Path, Plate Number, and Procedure allows appearing in that order from the beginning of the file on lines 3, 7, 8, 12, and 19 respectively. Any file that does not match this pattern should generate an error and the associated error message (see below).

These ‘blocks’ are repeated for each plate included in the file export, however the Gen5 adapter is only expected to support a single plate per export file. Any file that includes multiple plates should generate an error and the associated error message (see below).

Plate-Reader Aggregate Document

The plate-reader aggregate document represents the highest level of organization in the Plate-Reader ASM, and will contain the device system document - for recording details about the instrument, the data system document - for recording details about the data file, the plate-reader document - for recording data from each well of the plate, and where appropriate a calculated data aggregate document for recording calculated results from multiple measurements within calculated data document(s).

Device System Document

The device system document within the ASM is intended to capture metadata about the instrument from which the data was obtained.

The fields within this document can include:

  • asset management identifier
  • description
  • brand name
  • product manufacturer
  • device identifier (required)
  • model number (required)
  • equipment serial number
  • firmware version

The following device system document fields are provided in the Gen5 export as shown with their corresponding field in the Gen5 export file.

ASM device system document field

Gen5 field

model number

Reader Type

equipment serial number

Reader Serial Number

The Gen5 export does not include a field corresponding to device identifier, a required field in the ASM schema, so this can be recorded as “NA”.

Data System Document

The data system document within the ASM is intended to capture information about the instrument file, its location, source of generation, the adapter/converter used to generate the ASM file.

The fields within this document can include:

  • data system instance identifier
  • file name
  • UNC path
  • software name
  • software version
  • ASM converter name
  • ASM converter version

The data system instance identifier is intended to reference the computer/workstation/file-store on which the instrument file resides. This information is not included in the export, but if possible should be obtained by the adapter querying the identifier of the local system on which it is installed.

The file name is the name of the instrument file which the adapter is using as input.

The UNC path should be the complete file path of the instrument file.

The Gen5 export does not explicitly reference a software name but  “Gen5” can be entered as the default value, while software version is provided in the Software Version field in the Gen5 export.

The ASM converter name and ASM converter version refers to the adapter being described in this requirements document used to generate the ASM file from the instrument file. Therefore, the adapter should populate these fields with its own name and version.

Plate-Reader Document

The plate-reader document is an example of a technique document in the context of the Allotrope Simple Model, and is meant to represent each time the plate-reader technique is performed. In this framework the plate-reader technique is performed for each plate well (rather than plate) in the Gen5 export. This means that information from each well should be recorded in a separate plate-reader document.

The fields within this document can include:

  • measurement aggregate document (required)

Measurement Aggregate Document

The measurement aggregate document contains records of the measurements performed, each captured in a measurement document as metadata about the measurements performed. Note that the measurement aggregate document and constituent documents (e.g. measurement document, sample document) within the Plate-Reader ASM inherits generic fields from the core ASM schema as well as specific fields from the plate-reader ASM schema.

The fields within this document can include:

  • measurement time (required)
  • analytical method identifier
  • experimental method identifier
  • experiment type
  • container type
  • plate well count (required)
  • well volume
  • measurement document (required - per core schema)

The following measurement document fields are provided in the Gen5 export as shown with their corresponding field in the Gen5 export file.

ASM device system document field

Gen5 field

measurement time

Date and Time

analytical method identifier

Protocol File Path

experimental method identifier

Experiment File Path

For plate well count it is recommended that this value be derived from the product of the rows (designated by letters) x columns (designated by numbers) within the Layout block. While the entry Plate Type field often includes the number of wells within the plates, since the string returned in the Plate Type field is a descriptive name provided by the user of the type of plate it is not guaranteed to include the number of wells.

For container type, “well plate” can be used as the default value.

While the Date and Time are reported in separate fields, the measurement time should be recorded in the YYYY-MM-DD HH:MM:SS format, where time is formatted to 24-hour. Gen5 will include an AM / PM designation within the Time field. However, whether the Date is formatted MM/DD/YYYY or DD/MM/YYYY is determined by the Regional option defined by the Windows OS in which the file was generated.

The other optional fields within the measurement aggregate document including: experiment type, and well volume do not directly correspond to fields within the Gen5 export.

Measurement Document

The measurement document is an array of objects, each containing details about individual measurements recorded from each well within the SMP export.

The fields within this document can include:

  • measurement identifier (required)
  • sample document (required - per core schema)
  • device control aggregate document (required - per core schema)
  • compartment temperature (required)
  • measurement fields from one of:
  • ultraviolet-absorbance schema
  • fluorescence schema
  • luminescence schema

The measurement identifier is not derived from the instrument file and instead should be a universal unique identifier generated and added to the ASM file.

The compartment temperature is optionally included in the Gen5 export. When present the compartment temperature is included in the Results block in the Actual Temperature field. In cases of multiple measurements, i.e. measurements at different wavelengths the Actual Temperature field is repeated for each wavelength in the order it was performed. The wavelengths and the order in which they were acquired is listed in the Wavelengths field within the Read portion of the Header block.

Sample Document

The sample document contains details of the sample being assessed as part of the measurement.

The fields within this document can include:

  • description
  • sample identifier (required)
  • batch identifier
  • sample role type
  • written name
  • location identifier
  • well plate identifier
  • mass concentration

The Gen5 export contains different fields which map to the sample identifier depending on whether the user provides a sample identifier or not. The Layout block will always include a Well ID field which should be used as the sample identifier in the default case. However, in cases where a user explicitly provides a sample identifier, this will be reported in the Name field within the Layout block. In this latter case, while the Well ID field will also be present, this entry can be ignored in favor of the Name field. In cases in which the Layout block is not present, the concatenation of the well plate identifier and the location identifier can be used as the sample identifier (e.g. “Plate1 A1”).

The location identifier corresponds to the well of the plate in which the data was collected. The location identifier should be record as the concatenation of the Row and Column in the Results block, e.g. “A1”

The well plate identifier should be parsed from the file name, which is expected to begin with the naming convention “YYMMDD_HHMMSS_<Plate ID>_”. If the file name does not conform to this naming convention, then the well plate identifier can be taken from the Plate Number field in the Header block, which always follows the format of “Plate n”.

The remaining fields within the sample document do not correspond to fields provided in the Gen5 export.

Device Control Aggregate Document / Device Control Document

The device control document within the ASM is intended to record information about device settings under which each measurement was performed. The measurement document will always contain a device control aggregate document which in turn contains device control documents. The device control document is an array of objects, each recording details about a component of the instrument.

For plate readers such as those used in conjunction with Gen5 there will be an entry in the device control document for the detection module: absorbance, fluorescence, or luminescence. Additional entries to the device control document could be made for: the shaking configuration of the plate carrier, an integrated plate stacker, a gas control module, and/or a reagent injection module - however, details about these other module types do not need to be supported at this time.

Because the same device settings are typically used for the measurement of all samples within the dataset, it is expected that these values will be repeated across multiple measurement documents with the ASM file.

The fields within this document can include:

  • device identifier
  • device type (required)
  • detection type
  • product manufacturer
  • brand name
  • equipment serial number
  • model number
  • firmware version
  • shaking configuration description - used within a device control document for carrier shaking
  • detector distance setting (plate reader)
  • exposure duration setting
  • number of averages
  • detector gain setting
  • scan position setting (plate reader)
  • detector carriage speed setting
  • device control fields from one of:
  • ultraviolet-absorbance schema
  • detector wavelength setting
  • detector bandwidth setting
  • fluorescence schema
  • detector wavelength setting
  • detector bandwidth setting
  • wavelength filter cutoff setting
  • excitation wavelength setting
  • excitation bandwidth setting
  • luminescence schema
  • detector wavelength setting
  • detector bandwidth setting

The metadata related to device control is all recorded in the Read portion of the Procedure Details section within the Header block.

For device type “plate reader” can be used as the default value as it is a required field in the ASM.

The detection type will appear in the Read field as: “Absorbance Endpoint”, “Fluorescence Endpoint”, “Luminescence Endpoint”.  The string returned to the detection type field should be “Absorbance”, “Fluorescence”,  or “Luminescence” depending on the read type. This field also indicates which of the measurement schema (absorbance, fluorescence,  or luminescence) to use when creating the ASM JSON file.

Note: The Plate Reader ASM currently only supports Absorbance, Fluorescence, and Luminescence measurements. Any file that does not include “Absorbance”, “Fluorescence”,  “Luminescence” within the Read section should generate an error and the associated error message (see below).

Note: The Plate Reader ASM currently only supports endpoint measurements. Any file that does not Read section does not conform to the pattern of an Endpoint read (e.g. includes a “Kinetic” block) should generate an error and the associated error message (see below).

The remainder measurement document will include the absorbance, fluorescence, or luminescence field depending on detection type corresponding to Absorbance, Fluorescence, or Luminescence.

Measurement versus Calculated Values for Absorbance, Fluorescence, and Luminescence

Within the Gen5 export, both the measured data and calculated results are returned within the Results block as a plate matrix (row x column) again with every position in the matrix functioning as a list of entries.

The naming convention in the right margin of the Results block indicates whether the entry is a measurement or a calculated result. The entries for each well within the Results block are always ordered as measurement(s) followed by calculated result(s) - if present.

The measurement entries will always be indicated by the wavelength at which the measurement was performed, in the following format:

  • Absorbance - <detection wavelength>
  • example: 450
  • Fluorescent - <excitation wavelength>/<excitation bandwidth>, <emission wavelength>/<emission bandwidth> OR <excitation wavelength>, <emission wavelength>
  • example: 360/40, 460/40 OR 485, 528
  • Luminescence - <emission wavelength>/<emission bandwidth>  OR “Lum”
  • example: 460/40 OR Lum

Additionally measurements in Gen5 can be annotated with a user defined “Step Label”, when a Step Label is provided it will appear as a string before the measurement wavelength separated by a colon (:).

Absorbance measurements with Pathlength Correction

The use of Pathlength Correction presents an exception to the ordering pattern of ‘measured’ and ‘calculated values’.

Instances in which Pathlength Correction was used within an Absorbance measurement are indicated by the label “Pathlength Correction” appearing in the Read portion of Procedure Details followed by the two measurement wavelengths used in performing the correction, e.g. “Pathlength Correction: 977 / 900”

The results of the Pathlength Correction for each well are then reported within the Results block above the other measured and calculated values. The measured wavelengths will be labeled in the right margin of the Results block in the same format as other wavelength measurements and appended with either ‘[Test]’ or ‘[Ref]’. The measured values associated with these wavelengths should be recorded in separate measurement documents in the same fashion as all other measured wavelengths, while the ‘[Test]’ and ‘[Ref]’ strings can be ignored. Below these lines will be an entry labeled “Pathlength” which should be recorded in a calculated data document, with the measurement identifies of the ‘[Test]’ and ‘[Ref]’ wavelengths cited within the data source documents.  

For example:

  • FluorReadStepLabel:485/20,528/40

Lastly, any calculated results will appear below the measurement entries and are indicated by either: a string before the measurement - with no colon delimiter, a string before the Step Label:measurement - when a Step Label is present, or as string alone.

For example:

  • Blank 485/20,528/20 - where there is no Step Label
  • Blank FluorReadStepLabel:485/20,528/40 - where there is a Step Label
  • Ratio - where the calculated result is represented by the description alone

In the case where calculated results are present, these will not be recorded within the measurement document, but rather within the calculated data document (discussed below).

Processed Data Aggregate/Processed Data Document

The processed data document contains results data which is derived from measurement data with values that are dependent on user defined parameters or transformation. The parameters which determine the results within the processed data document are recorded in the data source document.

The fields within this document can include:

  • data source aggregate document
  • data source document
  • data source identifier (required)
  • data source feature

Calculated Data Aggregate / Calculated Data Document

Within the ASM, the calculated data document is used to record results that are derived from calculations performed using data across different measurement documents and/or other calculated data documents . A calculated data document differs from a processed data document in that a processed data document is a facet of a measurement document and contains results derived from transformation of the data within that measurement document.

While a calculated data document records calculated results, the adapter is not expected to perform any calculations, and rather parse calculated values which appear within the Gen5 file export.

All calculated data documents should be placed within the calculated data aggregate document within the plate reader aggregate document, regardless of whether the calculated data is derived from measurement documents from within the same plate reader document, or from measurement documents from different plate reader documents.

The fields within this document can include:

  • calculated data identifier
  • data source aggregate document
  • data source document
  • data source identifier
  • data source feature
  • calculated data name
  • calculation description
  • calculated result

Each value corresponding to a calculated result - as determined by the naming convention defined above - should be recorded within  a calculated data document.
