Revision 06

Background Section

The CTL ImmunoSpot adapter is intended to convert text file exports from CTL’s ImmunoSpot instrument software and generate a .json file corresponding to the plate-reader Allotrope Simple Model. ImmunoSpot software supports image-based fluorescent/non-fluorescent ELISpot counting, live/dead/apoptotic cell counting, and cytotoxicity experiments that can be performed on CTL’s line of ImmunoSpot analyzers. The adapter is expected to recognize the detection modality used to generate the data and generate the plate-reader ASM .json file using the optical-imaging measurement schema.

Note: Although ImmunoSpot software is able to perform several types of experiments (ie. ELISpot, Cytotoxicity, PBMC Cell Counting), this adapter only has examples of ELISpot experiment data as of this revision

Structure of ImmunoSpot File

The ImmumoSpot .txt export file is organized in (3) sections or “blocks”, each of which contain specific fields for mapping to the ASM. The (3) sections are not explicitly labeled, but correspond to “Device Information”, “Assay Information” and “Assay Data”.

  • Device Information
  • Block Start: Software version
  • Block End: Counting Window

  • Assay Information
  • Block Start: Assay
  • Block End: Edge Compensation Level

  • Assay Data

Plate-Reader Document

The plate-reader document is an example of a technique document in the context of the Allotrope Simple Model, and is meant to represent each time the plate-reader technique is performed. In this framework the plate-reader technique is performed for each plate well (rather than plate) in the instrument software export. This means that information from each well should be recorded in a separate plate-reader document.

Handling of Plate Well Count in CTL ImmunoSpot

The output text file of ImmunoSpot does not explicitly list the plate well count of experiment plates. Because there are several compatible options for plate well counts with ImmunoSpot (6, 12, 24, 48, 96, 384, 1536, Terasaki - 60 or 72), the adapter should then be expected to identify the first plate matrix within the output file and obtain the count of rows (alphabetical string value) and columns (numeric value) and multiply these two counts to obtain plate well count (ie. rows x columns).

For each output file there will be expected a string ‘Unprocessed Data’. From this string three lines below will be the line of column values. Two lines after will be the start of the first row string value ‘A’. The adapter should then read down the lines, obtaining a cumulative count until the string value ‘TNTC: Too numerous to count’ is identified.

Handling Image Files from ImmunoSpot

Output text files from the ImmunoSpot software do not explicitly list the associated image files/image path generated as part of an experiment. These image files are separate JPG files that follow the same naming convention as the text file with a concatenation of “_hr”[d][e] (high-resolution).

  • Text file name: “MOPA_C723655.txt”
  • Image file name: “MOPA_C723655_hr.jpg”

Additionally, images created by the ImmunoSpot software are not created on a per well basis but for an entire plate. The image aggregate document then should be recorded as a child of the measurement aggregate document - as opposed to the measurement document. The  experimental data identifier within the image document will be the image file name denoted by the naming convention above - with the same file name used for all wells in the plate.

Handling Image Features in ImmunoSpot

Image features will appear in the Data block of the output file with corresponding plate matrices per each image feature. All plate matrices should be recorded as image features as there is no measurement value that is associated with the ImmunoSpot output. The processed data aggregate document and child image feature document should be recorded at the measurement document level of the schema. *This will be important for mapping of location identifier to the respective image feature within the plate matrices.

The first plate matrix of each ImmunoSpot file (named in the file ‘Unprocessed Data’) corresponds to the ‘Spot Count’ image feature name. For handling of the values within this particular matrix, the adapter should expect numerical values. However, the instrument also assigns ‘TNTC’ (too numerous to count) and/or ‘?’ values to the plate wells. In the cases that non-numeric values are present, record the values as a “N/A” in the JSON.

The image feature name will appear at the line above the plate matrix and appear as a string value. The entire string value should be recorded for each of the image features including the characters enclosed in a parenthesis, if present.

The image feature result then will be the value that maps to the location identifier position of each of the plate matrices associated with an image feature name. Image feature results units should be recorded by default as “(unitless)”.

Calculated Data Aggregate / Calculated Data Document

Within the ASM, the calculated data document is used to record results that are derived from calculations performed using data across different measurement documents and/or other calculated data documents . A calculated data document differs from a processed data document in that a processed data document is a facet of a measurement document and contains results derived from transformation of the data within that measurement document.

While a calculated data document records calculated results, the adapter is not expected to perform any calculations, and rather parse calculated values which appear within the instrument software file export.

All calculated data documents should be placed within the calculated data aggregate document within the plate reader aggregate document, regardless of whether the calculated data is derived from measurement documents from within the same plate reader document, or from measurement documents from different plate reader documents.
