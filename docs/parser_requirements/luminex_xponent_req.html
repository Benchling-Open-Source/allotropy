Luminex xPONENT Multi-Analyte-Profiling Adapter Product Requirements

Revision 01

Date: 2024-01-03

The Luminex xPONENT multi-analyte-profiling adapter is intended to convert text file exports from Luminex xPONENT software and generate a .json file corresponding to the multi-analyte-profiling (MAP) Allotrope Simple Model (ASM). The Luminex xPONENT software is used in conjunction with the Luminex xMAP instruments which are used to perform multiplexed, bead-based, immunoassays for detecting a variety of antigens within a sample. The adapter is expected to parse data from the instrument and structure the data from each sample to the associated result fields.

The adapter is expected to produce the following output based on the provided input:

  • Input: luminex_xPONENT_example01.csv
  • Output: luminex_xPONENT_example01.json

Required User Actions

The required user actions are based on instructions provided in the FlexMap 3D User Manual (89-00002-00-622 Rev A).

The user should prepare samples and perform data acquisition and analysis following the manufacturer’s instructions.

Following the completion of an experiment, referred to as a “Batch”, the user should generate a .csv export of the data. To do so, the user should perform the following steps:

  • Navigate to the ‘Results’ page, and select the ‘Saved Batches’ tab
  • Choose the completed batch to export and click ‘Exp Results’
  • In the ‘Save As’ dialog box, choose the export destination for the .csv file
  • Click ‘Save’
  • Within the ‘Export Complete’ dialog box, click ‘OK’

Generation of the ASM file

The Allotrope ASM for multi-analyte-profiling has the following structure

multi analyte profiling aggregate document

⮑ device system document

⮑ calibration aggregate document

⮑ calibration document

⮑ calibration result aggregate document

⮑ calibration result document

⮑ data system document

⮑ multi analyte profiling Document

⮑ measurement aggregate document

⮑ measurement document(s)

⮑ sample document

⮑ device control aggregate document

⮑ device control document(s)

⮑ analyte aggregate document

⮑ analyte document(s)

⮑ calculated data aggregate document

⮑ calculated data document

⮑ data source aggregate document

⮑ data source document

The xPONENT export file has a number of ‘blocks’ which contain relevant measurement and metadata required for the ASM. The ‘blocks’ follow the pattern of:

Header - detailing device, file, experiment information, and calibration information

Calibration - detailing the time and results of instrument calibration

Samples - detailing the number of samples within the data set

Results - a set of tables containing results and additional metadata

  • Median
  • Net MFI
  • Count
  • Avg Net MFI
  • Units
  • Per Bead Count
  • Dilution Factor
  • Analysis Types
  • Audit Logs
  • Warnings/Errors

Multi Analyte Profiling Aggregate Document

The multi analyte profiling aggregate document will contain the Device System Document , Data System Document, multi analyte profiling document, and where appropriate a Calculated Data Aggregate Document.

Device System Document

The device system document within the ASM is intended to capture metadata about the instrument from which the data was obtained. Additionally the device system document can contain the calibration aggregate / calibration document capturing details about system calibration.

Within the xPONENT file export details for model number, and device serial number are all provided within the header block.

The model number is provided in the fourth column in the first row of the file - which starts with “Program”, while the equipment serial number is provided on the line starting with “SN”.

Calibration Document

The information to be recorded in calibration document is provided in the section beginning with the line “Most Recent Calibration and Verification Results”.

The entries on the following lines should be parsed are written to fields in the calibration document. The first column should be reported calibration name with the sting “Last” being removed.

The second column should be reported as the calibration report and calibration time. For example for the entry Passed 05/17/2023 09:25:11, “Passed” would be captured within the calibration report field, while the timestamp would be reported as the calibration time, with the timestamp changed to the YYYY-MM-DD T HH:MM:SS Z format.

All other values within the Device System Document can be ignored at this time.

Data System Document

The data system document within the ASM is intended to capture information about the instrument file, its location, source of generation, the adapter/converter used to generate the ASM file.

The data system instance identifier is intended to reference the computer/workstation/file-store on which the instrument file resides. This information is provided in the ComputerName field in the header.

The file name is the name of the instrument file which the adapter is using as input.

The UNC path should be the complete file path of the instrument file.

The software name and software version refer to the application that generated the instrument file. The software name is provided in the second column the first line of the xPONENT file starting with “Program”. In all cases the software name is expected to be “xPONENT”. The software version is provided in the Build field in the header.

The ASM converter name and ASM converter version refers to the the adapter being described in this requirements document used generate the ASM file from the instrument file. Therefore, the adapter should populate these fields with its own name and version.

Multi Analyte Profiling Document

The multi analyte profiling document is an example of a technique document in the Allotrope parlance, and meant to represent each time the MAP technique is perform. As the MAP technique is usually performed in multi-well plates, in this framework the MAP technique and multi analyte profiling document will correspond to each well in the data set. The individual wells are referred to as “Locations” within the xPONENT export, with each well appearing as a row in the Median, Count, Net MFI, and Avg Net MFI tables.

The multi analyte profiling document contains a Measurement Aggregate Document, details about analyst - the individual who executed the procedure, and the submitter - the individual who requested the procedure and submitted samples to the analyst.

The entry for analyst is recorded under the Operator in the header block. If no entry is provided, such that the field is empty - then this field can be ignored.

There is no entry corresponding to submitter within the xPONENT export.

Measurement Aggregate Document

The Measurement Aggregate Document contains details about the analytical method identifier, method version, experimental data identifier, experiment type, container type, plate well count, and details about the individual measurements within a Measurement Document array.

The analytical method identifier is provided in the ProtocolName, the method version is provided in the ProtocolVersion, and the experimental data identifier is provided in the Batch field, all within the header block.

The container type can be recorded as “well plate” as the default entry.

The plate well count is recorded as Type in the header block on the line beginning with ProtocolPlate

Measurement Document

The measurement document corresponds to a measurement performed on a sample with a particular set of device control settings. It is expected that the xPONENT export will only contain a single measurement per well, and thus a single measurement document per multi analyte profiling document.

The measurement document will contains the measurement identifier, the measurement time, the sample document, the device control aggregate/device control document, the assay bead count, the analyte aggregate/analyte document, and the error aggregate/error document.

The measurement identifier is not derived from the instrument file and instead should be a universal unique identifier generated and added to the ASM file.

The measurement time is recorded as the BatchStartTime field in the header block and should be recorded in a YYYY-MM-DD HH:MM:SS format.

The assay bead count recorded within the measurement document is intended to record the total number of assay beads detected within the sample. This value is provided in the Total Event column, the last most column in the Median table - although this column is also repeated in the Count, Net MFI, and Avg Net MFI tables as well, this is simply a repetition of the same data.

Sample Document

The sample document records metadata about the sample being measured. Of the fields included in the MAP ASM model, the xPONENT export only includes values that correspond to sample identifier and location identifier.

The sample identifier is provided under the Sample column in the Median table - and again repeated in the Count, Net MFI, and Avg Net MFI tables. Similarly, the location identifier is provided under the Location column. While the entries in the Location column are in the format of: 1(1,A1) - only the alphanumeric term (e.g. A1) should be recorded as the location identifier.

Device Control Document

The device control document records details about the instrument settings used during the measurement. For the xPONENT export details for device type, sample volume setting, dilution factor setting, detector gain setting, and minimum assay bead count setting can be recorded.

The device type is not reported in the export, but “multi analyte profiling analyzer” can be used as the default value.

The entry for sample volume and the detector gain setting are provided in the SampleVolume and the ProtocolReporterGain fields in the header.

The dilution factor setting is provided for each well (Location) in the Dilution Factor table.

The minimum bead count setting is reported in the Samples block in the Min Events field.

Analyte Document

MAP experiments are typically preformed to quantify multiple analytes - typically proteins - within a sample. Each analyte is detected using a specific bead functionalized to capture that analyte and quantify it’s abundance within the sample. Therefore, the data for analyte detected within the sample should be recorded using an analyte document within an analyte aggregate document.

Each analyte document can be used to capture details related to analyte identifier, analyte name, assay bead identifier, assay bead count, and fluorescence signal.

Within the xPONENT export, analytes are reported as columns in the Median, Count, Net MFI, and Avg Net MFI tables - with the rows corresponding to wells, and the different data types arrays as the different tables.

The analyte identifier , similar to the measurement identifier is not derived from the instrument file and instead should be a universal unique identifier generated and added to the ASM file.

The analyte name is provided as the column in the Median, Count, Net MFI, and Avg Net MFI tables - for all columns in between Sample and Total Events. The number of analytes should correspond to the value in the seventh column of the line in the header section beginning with ProtocolMicrosphere - in the field associated with Count.

The assay bead identifier is recorded in the Units table in the row labelled BeadID with the individual listed in the column under the heading associated with the analyte name.

The assay bead count is reported in the Count table in the row associated with the well and in the column under the heading associated with the analyte name.

The fluorescence is reported in the Median table in the row associated with the well and in the column under the heading associated with the analyte name. Since the reported value corresponds to the median fluorescence intensity (MFI) of all the beads corresponding to that particular analyte within that particular sample, fluoresence should include the property “has statistical datum role” : “median role”.

Error Document

The error document is intended to record the details of any errors reported by the instrument associated with a particular measurement, with the error documents placed within an error aggregate document.

The xPONENT reports errors within the Errors/Warning table, with each error associated with the well (location) with which it is associated. The Errors/Warning table has two columns Status and Message these two strings should be concatenated into a single “<status>:<warning>” string and recorded in the error field in the error document. The xPONENT export does not provide any further details about with analyte or property the error is associated with, so the error feature can be ignored at this time.

Calculated Data Document

The data reported within the Net MFI and Avg Net MFI would most appropriately be modeled using a calculated data document, however, the currently available test data does utilize the processes to perform these calculations - and as such in the test data these tables are simply a repetition of the results in the Median table. Therefore, at this time it is not necessary to format this data using calculated data documents.
