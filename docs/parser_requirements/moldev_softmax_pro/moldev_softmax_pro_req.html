Molecular Devices SoftMax Pro Plate Reader Adapter Requirements

Revision 04

Date: 2024-02-08

The Molecular Devices SoftMax Pro Plate Reader adapter is intended to convert text file exports from SoftMax Pro (SMP) software and generate a .json file corresponding to the plate-reader Allotrope Simple Model (ASM). The SoftMax Pro software supports the complete product line of plate readers manufactured by Molecular Devices, which included single-mode and multi-mode plate readers, meaning that the instrument are able to perform measurements using one (single) or more (multi) detection modality including: absorbance, fluorescence, or luminescence measurements. The plate-reader ASM support the recording of data from these different measurement modalities by referencing separate measurement schema for uv-absorbance, fluorescence, or luminescence. The adapter is expected to recognize the detection modality used to generate the data and generate the plate-reader ASM .json file using the appropriate measurement schema.

The adapter is expected to produce the following output based on the provided input:

Example 01 represents an endpoint absorbance experiment containing multiple plates. In addition to the absorbance measurements, the experiment contains a ‘Reduced’ data block and ‘Groups’ for which calculated resulted were generated. The Plate Block within the export is configured to contain Raw and Reduced data, and is formatted in the ‘Plate’ layout.

  • Input: MD_SMP_absorbance_endpoint_example01.txt
  • Output: MD_SMP_absorbance_endpoint_example01.json

Example 02 represents the same endpoint absorbance experiment in Example 01, including the ‘Reduced’ and ‘Groups’ data. In the Example 02 the data is exported in the Column rather than the Plate format. The output generated by the adapter is expected to be the same for both Examples 01 and 02.

  • Input: MD_SMP_absorbance_endpoint_example02.txt
  • Output: see example 01 output

Example 03 represents an endpoint luminescence experiment containing multiple plates. The data is exported in the ‘Column’ format. Additionally the Plate block includes both the Raw and Reduced data. This example does not contain ‘Group’ data.

  • Input: MD_SMP_luminescence_endpoint_example03.txt
  • Output: MD_SMP_luminescence_endpoint_example03.json

Generation of the ASM file

The Allotrope plate-reader ASM for Envision data has the following structure:

Plate-Reader Aggregate Document

⮑ Device System Document

⮑ Data System Document

⮑ Plate-Reader Document

⮑ Measurement Aggregate Document

⮑ Measurement Document(s)

⮑ Sample Document

⮑ Device Control Aggregate Document

⮑ Device Control Document(s)

⮑ Processed Data Aggregate Document

⮑ Processed Data Document

⮑ Data Processing Document

⮑ Calculated Data Aggregate Document

⮑ Calculated Data Document

⮑ Data Processing Document

The SMP export file has a number of ‘blocks’ which contain relevant measurement and metadata required for the ASM. There are (3) types of blocks: Note, Plate, and Group.

The total number if blocks in the file is designated at the top of the file by: ##BLOCKS= <#> , while ~End denotes the boundary of a block.

The type of block (Note, Plate, Group) is designated in position (1) of the first line of each block. In the case of Plate blocks this first line of the block - referred to hereafter as the header row, contains metadata that maps to other fields in the ASM.

The ‘blocks’ follow the pattern of:

Note:

-- Blocks of free text, though individual blocks may reference specific parameters, through use of accessors such as !InstrumentSerial for the instrument serial number. However, given the lack of constraints in the format of this block, the data from these does not need to be parsed.

Plate:

-- Raw Data: measurement values returned from reader

-- Reduced Data: a view of ‘well-level’ calculated result derived the Raw data, with the Reduced Data appearing in the block below the Raw Data

The Plate block can be exported in one of (2) formats: ‘Plate’ format in which the data is formatted in a matrix view of columns x rows. Or a ‘Column’ format in which each well is formatted as column with the well location (i.e. location identifier) as the column header. Which format is used is specified in position (4) of the Plate block header row, specified as PlateFormat - for the ‘Plate’ format, and TimeFormat - for the ‘Column’ format.

Note - There is a discrepancy between the SMP UI and the file export, the term Column is used in the UI while TimeFormat is used in the file export.

The orientation of these data depends on the type of acquisition.

For the ‘Plate’ format:

For measurements performed at multiple wavelengths the matrixes will appear as:

[wavelength 1][wavelength 2]...[wavelength n]

For kinetic measurements the matrixes will appear as:

[time 1]

[time 2]...

[time n]

For the ‘Column’ format:

For measurements performed at multiple wavelengths, the measurement at each wavelength will appear a new row in the ‘well’ column:

A01

wavelength 1

wavelength 2…

wavelength n

For kinetic measurements, the measurement at each time will also appear as a new row in the ‘well’ column:

A01

time 1

time 2…

time 3

Group:

-- Group Data: a table of aggregate results defined by the ‘group’ or wells

-- Group Columns: descriptors of the columns in the Group Data

-- Group Summary: Additional blocks containing results from user defined formula.

Plate-Reader Aggregate Document

The plate-reader aggregate document represents the highest level of organization in the Plate-Reader ASM, and will contain the device system document - for recording details about the instrument, the data system document - for recording details about the data file, the plate-reader document - for recording data from each well of the plate, and where appropriate a calculated data aggregate document for recording calculated results derived from multiple plate-reader documents (e.g. aggregated results from multiple wells).

Device System Document

The device system document within the ASM is intended to capture metadata about the instrument from which the data was obtained.

The fields within this document can include:

  • asset management identifier
  • description
  • brand name
  • product manufacturer
  • device identifier (required)
  • model number (required)
  • equipment serial number
  • firmware version

The SMP export does not include any instrument information by default, therefore the required fields of device identifier and model number can be recorded as “NA”.

Data System Document

The data system document within the ASM is intended to capture information about the instrument file, its location, source of generation, the adapter/converter used to generate the ASM file.

The fields within this document can include:

  • data system instance identifier
  • file name
  • UNC path
  • software name
  • software version
  • ASM converter name
  • ASM converter version

The data system instance identifier is intended to reference the computer/workstation/file-store on which the instrument file resides. This information is not included in the export, but if possible should be obtained by the adapter querying the identifier of the local system on which it is installed.

The file name is the name of the instrument file which the adapter is using as input.

The UNC path should be the complete file path of the instrument file.

The SMP export does not explicitly include information about software name or software version. For software name “SoftMax Pro” can be entered as the default value, while software version can be left unpopulated.

The ASM converter name and ASM converter version refers to the adapter being described in this requirements document used to generate the ASM file from the instrument file. Therefore, the adapter should populate these fields with its own name and version.

Plate-Reader Document

The plate-reader document is an example of a technique document in the context of the Allotrope Simple Model, and is meant to represent each time the plate-reader technique is performed. In this framework the plate-reader technique is performed for each plate well (rather than plate) in the SMP export. This means that information from each well should be recorded in a separate plate-reader document.

The fields within this document can include:

  • measurement aggregate document (required)

Measurement Aggregate Document

The measurement aggregate document contains records of the measurements performed, each captured in a measurement document as metadata about the measurements performed. Note that the measurement aggregate document and constituent documents (e.g. measurement document, sample document) within the Plate-Reader ASM inherits generic fields from the core ASM schema as well as specific fields from the plate-reader ASM schema.

The fields within this document can include:

  • measurement time (required)
  • analytical method identifier
  • experimental method identifier
  • experiment type
  • container type
  • plate well count (required)
  • well volume
  • measurement document (required - per core schema)

The measurement time is a required field, but no measurement time is returned in the SMP export, therefore this can be recorded as “null”.

The plate well count is provided in the header row of the Plate block. The position of this entry within the header row depends on the measurement type, at the (19) tab position for Absorbance and Luminescence measurement (20) tab position for Fluorescence measurements. This value will typically be 96 or 384.

For container type, “well plate” can be used as the default value.

The other optional fields within the measurement aggregate document including: analytical method identifier, experimental method identifier, experiment type, and well volume do not directly correspond to fields within the SMP export.

Measurement Document

The measurement document is an array of objects, each containing details about individual measurements recorded from each well within the SMP export.

The fields within this document can include:

  • measurement identifier (required)
  • sample document (required - per core schema)
  • device control aggregate document (required - per core schema)
  • compartment temperature (required)
  • measurement fields from one of:
  • ultraviolet-absorbance schema
  • fluorescence schema
  • luminescence schema

The measurement identifier is not derived from the instrument file and instead should be a universal unique identifier generated and added to the ASM file.

The compartment temperature is optionally included in the SMP export. When present the compartment temperature is included in the Plate block adjacent to the entry for well “A1”. When the compartment temperature is not present, this field can be excluded.

The absorbance / fluorescence / luminescence field would be populated by the entry corresponding to the well position (and wavelength - in the event of multiple measurements per well) within the Raw data section of Plate block.

In the event that the Raw data section of the Plate block is not included in the export OR if the Plate block is excluded entirely, the then following error message should be returned:

"The SoftMax Pro file is required to include either ‘Raw’ or ‘Both’ (Raw and Reduced) data for all plates"

Sample Document

The sample document contains details of the sample being assessed as part of the measurement.

The fields within this document can include:

  • description
  • sample identifier (required)
  • batch identifier
  • sample role type
  • written name
  • location identifier
  • well plate identifier
  • mass concentration

The SMP output does not include records that correspond to sample description, batch identifier, or written name.

The location identifier corresponds to the well of the plate in which the data was collected. The handling of the location identifier depends on the format in which the data is exported. If the file is exported in the ‘Column’ format, the well corresponds to the column header within the Plate block. While if the file is exported in the ‘Plate’ format then the location identifier needs to be inferred from the position in the matrix, with the row corresponding to the letter (A, B, C...) and column the number (1, 2, 3...).

The well plate identifier is provided in position (2) of header row of the Plate block.

The handling of the sample identifier field is dependent on which ‘blocks’ are present in the SMP export. For those files containing only Plate blocks, there is no field corresponding to the sample identifier provided. Since sample identifier is a required field in the ASM, for these cases, the concatenation of the well plate identifier and the location identifier can be used as the sample identifier (e.g. “Plate1 A1”). For files containing Group blocks, a Sample column is included within the Group Data, the entry in this column should be used as the sample identifier - associating it with the correct measurement document by using the Wells and PlateName columns within the Group Data.

Device Control Aggregate Document / Device Control Document

The device control document within the ASM is intended to record information about device settings under which each measurement was performed. The measurement document will always contain a device control aggregate document which in turn contains device control documents. The device control document is an array of objects, each recording details about a component of the instrument.

For plate readers such as those used in conjunction with SoftMax Pro there will be an entry in the device control document for the detection module: absorbance, fluorescence, or luminescence. Additional entries to the device control document could be made for: the shaking configuration of the plate carrier, an integrated plate stacker, a gas control module, and/or a reagent injection module - however, details about these other module types are not included within the SMP export.

Because the same device settings are typically used for the measurement of all samples within the dataset, it is expected that these values will be repeated across multiple measurement documents with the ASM file.

The fields within this document can include:

  • device identifier
  • device type (required)
  • detection type
  • product manufacturer
  • brand name
  • equipment serial number
  • model number
  • firmware version
  • shaking configuration description - used within a device control document for carrier shaking
  • detector distance setting (plate reader)
  • integration time
  • number of averages
  • detector gain setting
  • scan position setting (plate reader)
  • detector carriage speed setting
  • device control fields from one of:
  • ultraviolet-absorbance schema
  • detector wavelength setting
  • detector bandwidth setting
  • fluorescence schema
  • detector wavelength setting
  • detector bandwidth setting
  • wavelength filter cutoff setting
  • excitation wavelength setting
  • excitation bandwidth setting
  • luminescence schema
  • detector wavelength setting
  • detector bandwidth setting

The standard SMP export does not include any information about the instrument itself, therefore device identifier, product manufacturer, brand name, equipment serial number, model number, and firmware version, can all be ignored.

The metadata related to device control is all recorded in the header row of the Plate block.

For device type “plate reader” can be used as the default value as it is a required field in the ASM.

The detection type will appear in position (6) of the Plate block header row, and should correspond to Absorbance, Fluorescence, or Luminescence. This field also indicates which of the measurement schema to use when creating the ASM JSON file.

Calculated Data Aggregate / Calculated Data Document

Within the ASM, the calculated data document is used to record results that are derived from calculations performed using data across different measurement documents and/or other calculated data documents . A calculated data document differs from a processed data document in that a processed data document is a facet of a measurement document and contains results derived from transformation of the data within that measurement document.

While a calculated data document records calculated results, the adapter is not expected to perform any calculations, and rather parse calculated values which appear within the SMP file export.

All calculated data documents should be placed within the calculated data aggregate document within the plate reader aggregate document, regardless of whether the calculated data is derived from measurement documents from within the same plate reader document, or from measurement documents from different plate reader documents.

The fields within this document can include:

  • calculated data identifier
  • data source aggregate document
  • data source document
  • data source identifier
  • data source feature
  • calculated data name
  • calculation description
  • calculated result

The calculated data document includes a calculated data identifier, which similar to the measurement identifier is not derived from the instrument file and instead should be a universal unique identifier generated and added to the ASM file.

The calculated data document also includes a data source aggregate document/data source document which details the inputs to the calculation including: a data source identifier used to reference another measurement identifier or calculated data identifier, and a data source feature which is used to reference the field within the source document which was used as input to the calculation.

The calculated data document may contain a data processing document which will records needed to interpret how the calculation was performed. The terms used within the data processing document must conform to those previously defined in the Allotrope Foundation Ontology (AFO).

The calculated data document also includes a calculated data name, a name for the calculated result; optionally a calculation description to record how the calculation was performed; and a calculated result to record the calculated result.

Within the SMP export, both the Reduced Data from within the Plate block and the data within the Group block would be recorded within calculated data documents.

Plate Block - Reduced Data

For the Reduced Data the calculated data name will be “Reduced”. It is difficult to provide further context, since this calculation is user configurable within the SoftMax Pro UI and the SMP export does not provide any description of the calculation performed.

The data source identifier(s) should be the measurement identifier(s) of all measurement document(s) from the corresponding well in the same Plate block. For example, the Reduced value from well A1 would have as it’s data source identifier(s) any measurement identifier(s) from all measurement document(s) generated for A1.

The data source feature would be “absorbance”, “fluorescence”, or “luminescence” - depending on the measurement performed in the corresponding measurement documents.

Group Block

The Group blocks, the data is organized into tables in which each row represents a well and each column represents a calculated result - with the exception of columns defined below. For these data the calculated data name would correspond to the column header and the calculated result would be the column entry. The calculation description would be drawn from the entry under Formula Name for the corresponding column within the Group Columns portion of the Group block.

Only columns with numeric entries should be recorded as calculated results, additionally the following columns should not be recorded as calculated results :

  • Sample
  • Standard Value <unit> - where the unit string may vary
  • Wells
  • PlateName
  • R

Additionally, only Group blocks in which a Wells and WellPlateName column are present should be recorded as calculated data documents, since without this information it is not possible to confidently attribute the data to the source wells.

In the event that the event the the file contains Group block without the Wells and WellPlateName columns, the following warning message should be returned:

"The SoftMax Pro file is required to include Wells and WellPlateName columns for processing of Group data"

For columns within the Group block in which there is an entry for every well, the calculated data document should be constructed in a similar fashion those for the Reduced data, where the data source identifier(s) should be the measurement identifier(s) of all measurement document(s) from the corresponding well on that line - as defined by the Wells and WellPlateName column.

For columns within the Group block in which there is not an entry for each well, these data represent aggregations across a ‘set’ of wells - starting with row in which the value appears, and the subsequent blank rows. In these case, the calculated data document should be constructed where the data source identifier(s) should be the measurement identifier(s) of all measurement document(s) from that entire ‘set’ of wells.

In both cases, the data source feature would be “absorbance”, “fluorescence”, or “luminescence” - depending on the measurement performed in the corresponding measurement documents. It is expect that this data source feature would be common to all wells represents within the Group block, since measurements using more than one of these detection modes within the same experiment is uncommon.

While some of the data with Group blocks will be in reality be derived from other calculated values potentially in adjacent columns in the Group block, there is not enough information available to derived these relationships, so association with the original well measurements is the only inference that can be drawn.
