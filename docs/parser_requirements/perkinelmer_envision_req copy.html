Perkin Elmer Envision Plate Reader Adapter Requirements

Revision 04

Date: 2023-11-16

The Perkin Elmer Envision Plate Reader adapter is intended to convert text file exports from Envision software and generate a .json file corresponding to the plate-reader Allotrope Simple Model (ASM). The Envision is a multi-mode plate reader, meaning that it has the ability to perform absorbance, fluorescence, and luminescence measurements. The plate-reader ASM supports the recording of data from these different measurement modalities by referencing separate measurement schema for uv-absorbance, fluorescence, or luminescence. The adapter is expected to recognize the detection modality used to generate the data and generate the plate-reader ASM .json file using the appropriate measurement schema.

The adapter is expected to produce the following output based on the provided input:

  • Input: PE_Envision_fluorescence_example01.csv
  • Output: PE_Envision_fluorescence_example01_asm_wCalcDataDoc.json

Note: for demonstration purposes the output examples only contain the first sample present in the input examples

Additional Examples:

  • PE_Envision_example02_instrumentOutput.csv
  • PE_Envision_example03_instrumentOutput.csv

The ASM model schema are detailed in the following:

  • plate-reader ASM schema
  • ultraviolet-absorbance-point-detection ASM schema
  • fluorescence-point-detection ASM schema
  • luminescence-point-detection ASM schema

Instrument

Plate-Reader Aggregate Document

The plate-reader aggregate document represents the highest level of organization in the Plate-Reader ASM, and will contain the device system document - for recording details about the instrument, the data system document - for recording details about the data file, the plate-reader document - for recording data from each well of the plate, and where appropriate a calculated data aggregate document for recording calculated results derived from multiple plate-reader documents (e.g. aggregated results from multiple wells).

Device System Document

The device system document within the ASM is intended to capture metadata about the instrument from which the data was obtained.

The fields within this document can include:

  • asset management identifier
  • description
  • brand name
  • product manufacturer
  • device identifier (required)
  • model number (required)
  • equipment serial number
  • firmware version

The parameters for the device system document are captured within. the Instrument block, with equipment serial number in the Serial number field and device identifier within the Nickname field. As a default “EnVision” can be used as the model number

Data System Document

The data system document within the ASM is intended to capture information about the instrument file, its location, source of generation, the adapter/converter used to generate the ASM file.

The fields within this document can include:

  • data system instance identifier
  • file name
  • UNC path
  • software name
  • software version
  • ASM coverter name
  • ASM converter version

The data system instance identifier is intended to reference the computer/workstation/file-store on which the instrument file resides. This information is not included in the export, but if possible should be obtained by the adapter querying the identifier of the local system on which it is installed.

The file name is the name of the instrument file which the adapter is using as input.

The UNC path should be the complete file path of the instrument file.

The software name and software version refer to the application that generated the instrument file. For the EnVision, this information is provided on the last line of the .csv export, where the software name - EnVision Station, and software version is reported.

Example: “Exported with EnVision Workstation version 1.14.3049.1193

The ASM converter name and ASM converter version refers to the adapter being described in this requirements document used to generate the ASM file from the instrument file. Therefore, the adapter should populate these fields with its own name and version.

Plate-Reader Document

The plate-reader document is an example of a technique document in the context of the Allotrope Simple Model, and is meant to represent each time the plate-reader technique is performed. In this framework the plate-reader technique is performed for each plate well (rather than plate) in the EnVision export. This means that information from each well should be recorded in a separate plate-reader document.

The fields within this document can include:

  • measurement aggregate document (required)

Measurement Aggregate Document

The measurement aggregate document contains records of the measurements performed, each captured in a measurement document as metadata about the measurements performed. Note that the measurement aggregate document and constituent documents (e.g. measurement document, sample document) within the Plate-Reader ASM inherits generic fields from the core ASM schema as well as specific fields from the plate-reader ASM schema.

The fields within this document can include:

  • measurement time (required)
  • analytical method identifier
  • experimental method identifier
  • experiment type
  • container type
  • plate well count (required)
  • well volume
  • measurement document (required - per core schema)

The measurement time is provided under the Measurement date header within the Plate information block. For files containing data for multiple plates, there will be multiple entries for Measurement date, at this point only the earliest should be returned to the measurement time parameter.

The measurement time should be recorded in the YYYY-MM-DD HH:MM:SS format, where time is formatted to 24-hour.

The analytical method identifier should be obtained from the Protocol ID field and the experimental method identifier from the Assay ID field, both within the Basic assay information block.

For container type, “well plate” can be used as the default value.

The value for plate well count is provided in the Number of wells in plate field within the Plate type block.

Both experiment type and well volume are optional fields within the measurement aggregate document and do not directly correspond to fields within the EnVision export.

Measurement Document

The measurement document is an array of objects, each containing details about individual measurements recorded from each well within the EnVision export.

The fields within this document can include:

  • measurement identifier (required)
  • sample document (required - per core schema)
  • device control aggregate document (required - per core schema)
  • compartment temperature (required)
  • measurement fields from one of:
  • ultraviolet-absorbance schema
  • fluorescence schema
  • luminescence schema

The measurement identifier is not derived from the instrument file and instead should be a universal unique identifier generated and added to the ASM file.

Sample Document

The sample document contains details of the sample being assess as part of the measurement.

The fields within this document can include:

  • description
  • sample identifier (required)
  • batch identifier
  • sample role type
  • written name
  • location identifier (required) - one of:
  • well location identifier
  • vial location identifier
  • well plate identifier
  • mass concentration

The EnVision output does not include records that correspond to sample description, batch identifier, or written name.

The sample identifier corresponds to the entry within same position as the well of ‘plate-matrix’ within the Platemap block.

The well plate identifier corresponds to the plate barcode and is reported under the Barcode heading in the Plate information block. However, if no barcode is present, then the plate number list under the Plate heading should be returned as the well plate identifier. For example, an entry with (1) under Plate and no barcode, should have “Plate 1” listed as the well plate identifier in the ASM file.

The well location identifier needs to be inferred from the position of the measurement within the Plate block. The EnVision provides users with the option two export formats, one in which the row (A, B, C...) and column (1, 2, 3...) identifiers are included in the Plate bloc (see example 1) and another where these are not (see example 2). The measurement in the upper left most position a 96 well plate (8 rows, 12 columns) will correspond to a well location identifier of “A1”, while the measurement in the lower right most position will correspond to a well location identifier of “H12”. The number of rows and columns for a plate are reported within the Plate type block.

Device Control Aggregate Document / Device Control Document

The device control document within the ASM is intended to record information about device settings under which each measurement was perform. The measurement document will always contain a device control aggregate document which in turn contains device control document. The device control document is an array of objects, each recording details about a component of the instrument.

For a multi-mode plate reader such as the EnVision there will be an entry in the device control document for the detection module: absorbance, fluorescence, or luminescence. Additional entries to the device control document could be made for: the shaking configuration of the plate carrier, an integrated plate stacker, a gas control module, and/or a reagent injection module - however, details about these other module types are not included within the EnVision export.

Because the same device settings are typically used for the measurement of all samples within the dataset, it is expected that these values will be repeated across multiple measurement documents with the ASM file.

The fields within this document can include:

  • device identifier
  • device type (required)
  • detection type
  • product manufacturer
  • brand name
  • equipment serial number
  • model number
  • firmware version
  • shaking configuration description - used within a device control document for carrier shaking
  • detector distance setting (plate reader)
  • integration time
  • number of averages
  • detector gain setting
  • scan position setting (plate reader)
  • detector carriage speed setting
  • device control fields from one of:
  • ultraviolet-absorbance schema
  • detector wavelength setting
  • detector bandwidth setting
  • fluorescence schema
  • detector wavelength setting
  • detector bandwidth setting
  • wavelength filter cutoff setting
  • excitation wavelength setting
  • excitation bandwidth setting
  • luminescence schema
  • detector wavelength setting
  • detector bandwidth setting

For measurements performed with the ‘base’ EnVision instrument, a number of the field within the device control document can be ignored, since this information has already been captured within the device system document. These fields include: device identifier, product manufacturer, brand name, equipment serial number, model number, firmware version.

For device type “plate reader” can be used as the default value.

A challenge for interpreting the data within the EnVision export for assigning to measurement and device control fields within the ASM, is that unlike other multi-mode plate readers, the EnVision does not explicitly state the detection modality (absorbance, fluorescence, or luminescence) but instead relies on the concept of Labels for all detection setting.

Therefore the detection modality and thus the appropriate ASM fields to use must inferred from the Label field within the Plate information block and repeated again within the Label block.

In order to assign a detection type - measurements corresponding to absorbance will include the string “ABS” or “Absorbance” within the label. While luminescence measurements will include the string “LUM” or “Luminescence”. The remaining labels will correspond to fluorescence measurements and will include strings such as: “FI”, “FP”, “HTRF”, “LANCE”, “DELFIA”, “FRET”, “AlphaScreen” or “Fluorescence”.

The details for detector wavelength and bandwidth, as well as excitation wavelength and bandwidth are reference from the Label and the corresponding filters used within that Label.

For absorbance measurements, the detector wavelength is provided within the properties within the ASM correspond to the Excitation wavelength field within the Label.

For fluorescence measurements, the detector and excitation properties are provided within the Filters section within the Labels block. The Filter Type field will specify whether the filter is Excitation or Emission, corresponding to excitation and detector within the ASM. The values for wavelength and bandwidth are found within the Description field of the individual filter, with CWL referencing the wavelength and BW referencing the bandwidth.

As shown in example 1, a single Label might be measured with (2) detectors as listed under the MeasInfo heading, indicated by De=1st, De=2nd. In this case, the results data for each plate will be reported in the order of the detectors in the Plate block, and these two results should be reported in separate measurement documents. Therefore the corresponding detector wavelength and bandwidth values for De=1st : Ems. filter and De=2nd : 2nd ems. filter should be used in each device control document.

The luminescence measurements will often not include a record for detector wavelength of detector bandwidth as luminescence measurements typically record broad spectrum light. If this information is present, it will be recorded as part of the Emission filter Description.

Remainder of Measurement Document

The remainder of the measurement document includes the compartment temperature; the mode specific measurement: absorbance, fluorescence, or luminescence; and (if necessary) a processed aggregate / processed data document.

The compartment temperature is reported in the Chamber temperature at start field within the Plate information block.

The mode specific measurement: absorbance, fluorescence, or luminescence is used depending on the acquisition modality used.

Lastly, in the ASM framework a processed data document is used to capture ‘processed data’, data which derives from a measurement, but whose values is determined by a user defined parameter - such that by change the value of that user defined parameter would return a different value for that result. While the EnVision may produce results that would qualify as ‘processed data’, none of the available data sets contain records that should be handled in this fashion.

Calculated Data Aggregate / Calculated Data Document

Within the ASM, the calculated data document is used to record results that are derived from calculations performed using data across different measurement documents and/or other calculated data documents . A calculated data document differs from a processed data document in that a processed data document is a facet of a measurement document and contains results derived from transformation of the data within that measurement document.

While a calculated data document records calculated results, the adapter is not expected to perform any calculations, and rather parse calculated values which appear within the EnVision file export.

All calculated data documents should be placed within the calculated data aggregate document within the plate reader aggregate document, regardless of whether the calculated data is derived from measurement documents from within the same plate reader document, or from measurement documents from different plate reader documents.

The fields within this document can include:

  • calculated data identifier
  • data source aggregate document
  • data source document
  • data source identifier
  • data source feature
  • calculated data name
  • calculation description
  • calculated result

The calculated data document includes a calculated data identifier, which similar to the measurement identifier is not derived from the instrument file and instead should be a universal unique identifier generated and added to the ASM file.

The calculated data document also includes a data source aggregate document/data source document which details the inputs to the calculation including: a data source identifier used to reference another measurement identifier or calculated data identifier, and a data source feature which is used to reference the field within the source document which was used as input to the calculation.

The calculated data document may contain a data processing document which will records needed to interpret how the calculation was performed. The terms used within the data processing document must confirm to those previously defined in the Allotrope Foundation Ontology (AFO).

The calculated data document also includes a calculated data name, a name for the calculated result; optionally a calculation description to record how the calculation was performed; and a calculated result to record the calculated result.

Within the EnVision export, calculated data is reported within the Calculated results block, while the Formula field within the Plate information block describes the calculation.

To capture this information within a calculated data document, the entry in the Formula field should be recorded in the calculation description, while the value to the left of the ‘=’ in the Formula should be used as the calculated data name. While the calculated result would value within the well of ‘plate-matrix’ of the Calculated results block, and the data source identifier would correspond to the measurement identifier(s) of the well in the same plate.
