ASM Adapter Template Instructions

Revision 1

Date: 2024-03-05

Purpose:

This document is intended to provide instructions on using the ASM adapter templates, which are used for drafting requirements for converting laboratory instrument exports to the corresponding Allotrope Simple Model (ASM) data model.

Questions:

Please post any questions about these instructions or those that arise during the generation of adapter requirements to Issues >

Template Formats:

The ASM adapter templates have the following format:

  • Title - this should be updated to reflect the instrument software being described in the requirements
  • Current Revision / Date of Revision
  • Generation of ASM file - including overall structure of the ASM JSON
  • Tables describing the ASM documents and child documents
  • Change Log

The cells within the table colored in gray are meant to be instructive and/or contain pre-determined parameters and should not be modified by the individual using the template.

Table Columns:

  • ‘ASM Field’ refers to the document or keyword used within the ASM JSON schema

  • ‘Where is the value derived?’ is meant to inform both the author using the template and the software developer who will be using the requirements how the information should be obtained. The options in this column include:
  • Child Document(s) - Indicating that this is a child document of the current document and will be described elsewhere in the requirements. The ASM uses “aggregate documents” for those cases where there may be multiple instances of the “child document”.
  • ID generated by Adapter - Indicating that this field should be populated by a UUID generated by the adapter itself.
  • Instrument File - Indicating that this is information that the adapter is expected to parse from the instrument file.
  • Hardcoded Value - Indicating that this field can be populated with a hardcoded value. The value itself must be provided by the author of the adapter requirements.

    Hardcoded values should only be used in cases where there is high confidence in what the entry should be. In the event that the appropriate information for the field is not provided by the instrument export and the field is not required, it is recommended to exclude the field entirely from the JSON file generated rather than relying on hardcoded values.
  • Value generated by the Adapter - Indicating that this is value - typically an index value or similar - which should be generated by the adapter and reported in this field.
  • ID referenced by Adapter - Indicating that this field should be populated with a UUID that is used elsewhere in the ASM file - in order to establish a link between two sections of the JSON file.

    A common use of this is the     Data Source Identifier field within a Calculated Data Document, which contains a UUID Measurement Identifier field to indicate the data from that measurement is the source of the information being reported within the Calculated Data Document.
  • Keyword referenced by Adapter - Indicating that this field should contain a keyword used elsewhere in the ASM file.

    Similar to the     ID reference by Adapter this is used to provide contextual information about which facet of a reference document cited in a Data Source Identifier is relevant.

  • Required’ column is meant to indicate whether the associated field is a required field of the JSON schema.  

In the event that the information for a required field is not available within the instrument export, the author of the adapter requirements will need to define whether (i) the generation of the ASM JSON should produce an error, (ii) the field should be populated with with a non-numeric constant ‘NaN’ - in the context of a numeric field, (iii) the field should be populated with with an “NA” - in the context of a string field, of (iv) populated with some other hardcoded value.

  • The [Instrument Software Field] is meant to be populated by the requirements author with the corresponding field in the instrument export where the data should be derived.

The text “ [Instrument Software] ” should be replaced with the name of the software being described

  • The Details column is meant to be populated by the author with more context about the data mapping where necessary.

The Details column may contain normal text which is intended to serve as pre-populated context. Alternatively, the Details column may contain italic text which is intended to serve as a note to the requirements author - the author should update the content of this cell in the table and/or delete the text entirely.

In addition to the Details column the author should use the document outside the table to provide further details about interpreting the data within the instrument export, annotated screenshots, and other behavior expected by the adapter.

Unused Rows:

The requirements author should strike through the contents of any rows within the ASM document table that refer to non-required fields for which corresponding information is not provided in the instrument export. This helps make it easier for the software developer to know that this field need not be included in the ASM JSON file generated by the adapter.

Remainder of Template Document:

Note to Author:

Throughout the template document, sections labeled “Note to Author” will appear in bold italic. These are intended to provide instructions to the author using the template and should be deleted in the final requirements document produced from the template. 

Typesetting:

In order to make it clearer to the reader when the author is referring to a field within the instrument export or a field within the ASM JSON, throughout the requirements document fields within the instrument export, these should be set in Arial Italic, while fields within the ASM JSON these should be set in Robotic Mono.

The Paint Format tool within Google Docs can be used to apply the same typesetting repeatedly to text throughout the document.
